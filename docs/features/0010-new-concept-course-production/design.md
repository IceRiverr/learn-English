# Feature 0010：新概念英语四册课程制作与上线

状态：已实现并部署

## 1. 结论

本次工作把《新概念英语》第一至第四册制作成应用可直接使用的双语听力课程，并完成课程目录、播放器接入、静态资源构建和生产部署。

生产地址：<https://learn.iceriver.cc>

最终上线内容如下：

| 册数 | 音频/课程文件数 | 英文字幕段数 | 简体中文译文 | 说明 |
| --- | ---: | ---: | --- | --- |
| 第一册 | 72 | 1500 | 1500 | 每个音频包含一课正文及下一课练习，共覆盖 144 课 |
| 第二册 | 96 | 1430 | 1430 | 每课一个音频 |
| 第三册 | 60 | 1570 | 1570 | 每课一个音频 |
| 第四册 | 48 | 1088 | 1088 | 每课一个音频 |
| 合计 | 276 | 5588 | 5588 | 全部课程都有 `zh-Hans` 译文 |

课程音频和 JSON 作为静态资源随站点发布。MP3 不进入 Git，也不由 Service Worker 预缓存；用户仍通过播放器现有的下载功能按课程保存音频，以免首次安装 PWA 时下载约 596 MB 的全部音频。

## 2. 背景与目标

仓库原来只有少量示例课程。用户希望先用第四册第 5 课验证完整流程，再把第三、第四册扩展完成，最后按相同标准完成第一、第二册并上线。

本 Feature 的目标是：

1. 从本地 MP3 生成带时间轴的英文字幕。
2. 使用 Codex 直接把英文字幕翻译为简体中文。
3. 把英文和中文合并进单一课程 JSON，并进行可重复的完整性校验。
4. 将四册课程自动生成到应用课程目录中。
5. 只发布明确允许上线的公共资源，避免把 `public` 下的其他材料意外带入构建产物。
6. 完成生产部署，并验证 JSON、MP3、字幕和播放器行为。

本次没有增加后台、用户权限或运行时翻译服务。双语内容是在制作阶段生成的静态数据。

## 3. 目录与数据格式

四册内容分别位于：

```text
public/新概念/新概念1-美音/
public/新概念/新概念2-美音/
public/新概念/新概念3-美音/
public/新概念/新概念4-美音/
```

每个 MP3 对应一个同名 JSON。课程 JSON 继续使用现有版本 1 格式：

```json
{
  "version": 1,
  "course": {
    "id": "nce-3-001",
    "title": "Lesson 1: A Puma at Large",
    "audioFilename": "01－A Puma at large.mp3",
    "duration": 213.5,
    "language": "en"
  },
  "segments": [
    {
      "id": "segment-0001",
      "start": 0.0,
      "end": 4.2,
      "text": "Pumas are large, cat-like animals...",
      "translations": {
        "zh-Hans": "美洲狮是一种体形似猫的大型动物……"
      }
    }
  ]
}
```

重要约束：

- `course.id` 是下载、课程信息和学习进度的稳定标识，标题调整时不得改变。
- `audioFilename` 必须和实际 MP3 文件名完全一致。
- 字幕段 ID 在翻译前后保持不变。
- 字幕时间不能重叠，`end` 必须大于 `start`，且不能超过音频时长的允许误差。
- 英文始终存在；中文放在 `translations["zh-Hans"]`，播放器仍兼容没有翻译的旧课程。

## 4. 英文字幕制作流程

### 4.1 本地转写

英文字幕使用本地 `faster-whisper` 的 `small.en` 模型生成，主要参数为：

- CPU `int8` 推理；
- word timestamps；
- beam size 5；
- VAD；
- 句子最长约 16 秒；
- 静音间隔达到约 2.5 秒时优先断句。

转写原始结果和制作中间文件保存在被 Git 忽略的 `.translation-work` 中，不作为应用资源发布。课程 JSON 只保留播放器需要的最终字幕。

第一、第二册的文件组织有所不同：

- 第一册有 72 个 MP3，文件名类似 `001&002－...`。奇数课是正式课文，紧随其后的偶数课是练习，因此一个音频覆盖两课，但在课程目录中只形成一个播放项。
- 第二册有 96 个 MP3，每个文件对应一课。
- 第三册有 60 个 MP3，第四册有 48 个 MP3。

### 4.2 英文字幕校对

自动转写完成后进行以下检查：

1. 对人名、地名、专有名词、同音词和明显语法错误进行人工语义校正。
2. 检查音频末尾是否混入来源说明或重复片段；只清理字幕，不修改 MP3。
3. 对 VAD 漏掉的短句，使用关闭 VAD 的转写结果辅助恢复。
4. 检查重复段、残句、缺句和相邻时间轴重叠。
5. 英文或时间轴发生变化后，重新准备翻译清单，禁止继续复用旧清单。

字幕校对仍是内容制作中最需要人工判断的环节。自动校验可以发现结构问题，但不能证明每个单词的语义都正确。

## 5. Codex 双语翻译流程

翻译统一使用 `scripts/translation-workflow.mjs`，而不是直接批量重写大型课程 JSON。

### 5.1 Prepare

```powershell
node scripts/translation-workflow.mjs prepare <course-json> <work-dir> 10
```

`prepare` 按约 10 分钟的时间窗口拆分字幕，并为每块生成：

- 待翻译的 `targetSegments`；
- 前后三条只读上下文；
- 可复用术语表；
- 预期输出文件名和字幕 ID 清单；
- 源课程 SHA-256 摘要。

摘要覆盖课程元数据、标题、英文文本、字幕 ID 和时间轴。只要这些内容发生变化，旧翻译任务就会失效，必须重新执行 `prepare`。

### 5.2 Codex 翻译

Codex 直接读取分块输入，把 `targetSegments` 翻译成自然的简体中文，输出严格保持以下结构：

```json
{
  "language": "zh-Hans",
  "translations": {
    "segment-0001": "……"
  }
}
```

不得增删、合并、拆分或重命名字幕 ID。上下文只用于消歧，不写入输出。

批量制作时，可以由多个 Codex 子任务处理互不重叠的课程或分块。每个子任务只写自己的输出文件，汇总前再进行独立复核，避免多人同时修改同一个大型 JSON。

### 5.3 Merge 与 Validate

```powershell
node scripts/translation-workflow.mjs merge <course-json> <work-dir> zh-Hans en
node scripts/translation-workflow.mjs validate <course-json> <work-dir> zh-Hans en
```

`merge` 在摘要匹配时，按字幕 ID 把翻译写入最终课程 JSON。

本次增强了 `validate`：它不仅检查译文是否非空，还会重新读取所有分块输出，逐条确认最终 JSON 中的译文与已审核分块完全一致。校验同时保证：

- 分块没有缺失或多余 ID；
- 每个 ID 只出现一次；
- 全部英文字幕都得到译文；
- 英文源、字幕 ID 和时间轴没有在翻译后被改动；
- `course.language` 为 `en`；
- 合并结果没有被后续操作意外覆盖。

## 6. 标题规范

最初的标题包含较长的重复前缀，例如“新概念英语第三册 Lesson …”。上线前统一缩短 JSON 中的 `course.title`，课程目录直接使用 JSON 标题，不在界面再次拼接册名。

标题规则为：

- 第一册：使用音频中奇数正式课的规范标题，例如 `Lesson 1: Excuse me!`、`Lesson 29: Come in, Amy.`、`Lesson 99: Ow!`。不能写成 `Lessons 1–2`，因为偶数课是练习而不是第二篇课文。
- 第二至第四册：使用 `Lesson N: Title`。
- 不保留“新概念英语第几册”等重复前缀；册名由课程分组标题表达。
- 标题标点遵循教材语义，不根据文件名机械猜测。

为保证批量修改可复现，增加了：

- `scripts/shorten-new-concept-titles.mjs`：规范第三、第四册标题；
- `scripts/normalize-new-concept-1-titles.mjs`：使用第一册 72 个奇数正式课的规范标题表。

标题属于翻译清单摘要的一部分。修改标题后必须重新 `prepare`，再确认已有翻译输出与新清单同步。

## 7. 应用代码调整

### 7.1 自动生成课程目录

新增 `scripts/generate-new-concept-lessons.mjs`，扫描四册 JSON 和 MP3，校验完成后生成 `src/lessons.ts`。

生成器使用 Zod 校验外部 JSON，并额外检查：

- 课程语言必须为 `en`；
- 字幕 ID 不重复；
- 时间轴有效且不重叠；
- 字幕不超过课程时长；
- 每段都有非空的 `zh-Hans`；
- JSON 引用的 MP3 实际存在；
- MP3 文件大小可用于课程列表展示。

`src/lessons.ts` 是生成文件，不应手工维护。课程内容变化后重新运行生成器即可，避免在 `App.tsx` 中维护数百条硬编码记录。

生成后的目录共有 279 项：276 个新概念课程，加上原有 3 个其他课程。

### 7.2 课程列表界面

`src/App.tsx` 改为读取生成的 `lessonCollections` 和 `lessons`，并按以下分组显示：

1. 新概念英语第一册；
2. 新概念英语第二册；
3. 新概念英语第三册；
4. 新概念英语第四册；
5. 其他课程。

每个分组使用可折叠的 `<details>`，标题旁显示课程数量。`src/app.css` 增加了分组样式和移动端触控尺寸，窄屏仍保持可操作性。

播放器继续使用现有的双语能力：

- 英文/双语切换；
- 当前字幕和完整字幕视图；
- `zh-Hans` 缺失时仍能播放英文；
- 本地已保存旧课程但远端新增翻译时，刷新课程元数据而不破坏已下载音频和进度。

## 8. 构建与 PWA 资源策略

直接使用 Vite 默认的 `public` 复制行为会把 `public` 下所有本地材料都复制到 `dist`。为控制发布边界，`vite.config.ts` 调整为：

```ts
build: {
  copyPublicDir: false
}
```

并增加构建插件，只复制：

- `icon.svg`；
- `samples`；
- 新概念第一至第四册目录。

这样可以防止未完成、未授权或仅供本地使用的其他材料被意外上线。

Workbox 仍只预缓存 HTML、JavaScript、CSS 和 SVG 等应用外壳，不预缓存 MP3，也不预缓存 `samples`。课程音频由浏览器按需请求，用户主动下载后再进入 OPFS 或 IndexedDB。

## 9. 部署流程

标准生产部署命令为：

```powershell
pnpm deploy
```

`scripts/deploy.ps1` 的流程是：

1. 优先使用 Codex 随附的稳定 Node/pnpm 运行时，规避本机 Node 版本导致的 Vite 构建崩溃。
2. 执行完整生产构建，构建失败立即停止。
3. 通过 Windows OpenSSH 连接服务器。
4. 只创建和写入 `/var/www/learn.iceriver.cc`。
5. 把 `dist` 上传到生产目录。
6. 恢复调用前的进程 `PATH`。

部署目标不得改为 `/var/www/iceriver.cc`。

首次上线四册时上传完整资源。后续仅修改 JSON 或应用外壳时，可以采用经核对的选择性上传，避免重复传输数百 MB 的未变化 MP3；选择性上传前后必须核对远端目标路径，且不得在站点目录遗留临时目录。

## 11. 日常维护流程

新增或修正一门课程时，按以下顺序操作：

1. 校对英文、字幕 ID 和时间轴。
2. 执行 `translation-workflow.mjs prepare`。
3. 由 Codex 翻译对应分块，并做语义复核。
4. 执行 `merge` 和 `validate`。
5. 运行 `node scripts/generate-new-concept-lessons.mjs`。
6. 运行 `pnpm build`。
7. 运行 `git diff --check`。
8. 只有用户明确要求生产部署时，才运行 `pnpm deploy`。
9. 部署后检查课程 JSON、MP3 Range 请求和播放器实际播放。

如果只修改标题，也要从第 2 步重新开始，因为标题已进入源摘要。

## 13. 非目标与已知限制

- 本 Feature 不处理音频版权和再发布授权流程；公开新增课程前仍需单独确认来源、署名和授权状态。
- 不提供后台管理、用户权限、在线编辑器或翻译服务。
- 不把 MP3 或 `.translation-work` 中间文件加入 Git。
- 不改变播放器现有的逐句循环、存储和离线架构。
- 自动转写和机器辅助翻译仍可能存在个别语义错误；后续内容修订应走同一套 prepare、merge、validate 流程。
- 单词点击翻译等后续功能不属于本 Feature。
