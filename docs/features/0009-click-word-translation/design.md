# Feature 0009：点击英文单词显示翻译

状态：暂存，暂不开发

## 1. 结论

本 Feature 暂不实现。

iPhone 已提供系统级选词和词典查询能力。当前产品的主要目标平台是 iPhone Safari 和 iOS 主屏幕 Web App，
现阶段没有足够收益在播放器内重复建设分词、词形还原、英汉词典、弹层和离线词典存储。

开发计划中的 F0009 可以保留为候选需求。以后只有在系统词典无法满足实际学习需求，或需要跨平台一致体验时，
再重新评估。

## 2. 原始需求

规划来源为 `docs/general/英语听力-开发计划.md`：

```text
F0009，增加点击单词，显示翻译。
```

最初设想是在当前句和全文视图中点击英文单词，显示简体中文含义，并讨论了以下范围：

- 可点击范围；
- 英文分词、标点、撇号和连字符；
- 词形还原；
- 单词和短语释义来源；
- 上下文释义；
- 移动端弹层交互；
- 发音；
- 离线缓存、隐私和失败降级。

讨论后已明确首版即使恢复，也不需要发音，只考虑显示单词或短语的中文意思。

## 3. 与当前实现的关系

当前播放器有两个互斥正文视图：

- **当前句**：英文是普通文本；
- **全文**：整条字幕是按钮，点击后 seek 到该句并自动播放。

如果以后恢复本 Feature，全文视图存在明确交互冲突：英文单词点击要用于查义，而字幕行点击当前用于 seek。
不能在现有字幕行按钮中直接嵌套单词按钮。

可行调整是把全文字幕行改为非交互容器，由左侧时间或播放图标提供不小于 44 CSS 像素的“播放此句”按钮，
英文正文专门用于查词。Feature 0009 若采用该方案，需要明确覆盖 Feature 0004 中“点击英文区域跳转”的旧行为，
但不得改变 Feature 0006 的循环重置语义。

词义弹层本身不应 seek、暂停、切换循环目标或修改播放进度。切换课程、句子或正文视图时应关闭弹层。

## 4. 讨论过的方案

### 4.1 课程逐词注释

在课程制作阶段为每个单词或短语生成字符范围和上下文中文释义，例如：

```json
{
  "text": "They are running a small restaurant.",
  "wordTranslations": [
    {
      "charStart": 9,
      "charEnd": 16,
      "translation": "经营"
    }
  ]
}
```

优点：

- 释义可以结合当前句消歧；
- 完全离线；
- 点击时没有网络请求；
- 运行时代码直接。

缺点：

- 每门课程都要增加逐词内容制作流程；
- 字幕修改后必须重新生成并校验字符范围；
- 释义会大量重复；
- 实际上形成了新的课程内容维护负担。

用户不希望自行维护词义，因此不采用该方案。

### 4.2 在应用中实现小型词典

曾讨论增加课程级 `lexicon`，保存 headword、lemma、词性、词形和多个义项，再由每条字幕保存必要的上下文
消歧和短语范围。

该方案能够减少重复数据，但本质上是在应用里实现一个小型词典系统，需要处理：

- `running`、`ran` 等词形到 `run` 的映射；
- 同形异义和词性；
- 短语最长匹配；
- 通用义与本句义；
- 词典版本、校验和更新。

对于“点击后只看中文意思”的需求，这个范围过重，因此不采用。

### 4.3 使用通用离线词典

比自行维护词义更合理的方向是采用有明确许可的通用英汉词典。

讨论中优先评估了 [ECDICT](https://github.com/skywind3000/ecdict)：它提供英汉词典数据、词形资料以及
CSV、SQLite、StarDict 等形式，其生成的词库也可用于欧路、MDict 和 GoldenDict。仓库标注为 MIT License，
但其说明同时提到数据经过多年从多种资料整理而来；正式公开再分发前仍需确认并保留许可证、来源和必要声明。

[FreeDict](https://freedict.org/downloads/) 也提供可下载的英中词典，许可来源相对规整，但词条覆盖量较小。

如果以后采用通用词典，推荐把开放词典作为只读上游数据，在构建阶段自动提取内置课程中实际出现的词条，
生成每门课程的精简词典。不建议把完整大词典直接加入应用外壳，也不建议在点击时调用在线词典 API。

这种方案不需要人工维护释义，但只能展示通用词典义，不能保证自动选择当前句最准确的义项。

### 4.4 导入欧路兼容词典

欧路官方说明支持 MDict 的 `.mdx` / `.mdd`、灵格斯 `.l2d` 和 Babylon `.bgl` 等扩充词库格式：

- <https://docs.eudic.net/1/shi-yong-zhi-yin/tian-jia-kuo-chong-ci-ku>

但“欧路可以导入”只表示文件格式兼容，不代表词典内容拥有再发布许可。网上流传的商业词典 MDX 文件不得在
没有明确授权时打包进本项目。

iOS 应用沙箱也意味着 PWA 不能直接读取已经安装在欧路 App 中的词典。若允许用户另行导入自己的 MDX，项目
仍需实现不同 MDX 版本的解压、索引、MDD 资源、HTML 净化、大文件存储和 iPhone 随机查询。这会把本 Feature
扩大成浏览器词典引擎，因此不建议作为 F0009 的实现方式。

### 4.5 使用 iPhone 系统词典

最终选择是继续使用 iPhone 已有的系统选词和词典查询能力。

优点：

- 不需要维护或再分发词典内容；
- 不增加应用体积和本地存储；
- 不引入在线翻译服务、密钥、费用和额外隐私风险；
- 不需要改变当前字幕点击和播放器循环行为；
- 不增加依赖、数据库字段或 Service Worker 缓存范围。

局限：

- 体验由操作系统和用户安装的词典决定；
- 系统选词通常需要长按，不是应用内单击；
- 不同桌面浏览器和非 iOS 平台体验不一致；
- 应用无法控制释义语言、内容和弹层样式。

当前这些局限不足以证明自建功能的开发和维护成本合理。

## 5. 如果以后恢复

恢复设计前应先通过真实 iPhone 验证以下问题：

1. Safari 和主屏幕 Web App 中能否稳定选择字幕单词并打开所需英汉词典；
2. 单词按钮化是否会明显优于系统长按查词；
3. 是否必须支持自定义导入课程；
4. 是否需要上下文义，还是接受通用词典列出的多个义项；
5. 是否愿意承担词典数据许可、版本和离线存储维护；
6. 全文字幕 seek 是否可以改为独立的时间或播放按钮。

若仍决定实现，建议优先顺序为：

1. 使用许可明确的开放英汉词典；
2. 构建阶段生成内置课程的精简词典；
3. 运行时只做英文分词、最长短语匹配和只读释义展示；
4. 词典缺失或损坏时继续正常播放字幕；
5. 不实现发音、实时 AI 解释、生词本、查词历史或任意 MDX 导入。
