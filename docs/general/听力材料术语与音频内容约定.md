# 听力材料术语与音频内容约定

状态：设计约定，Feature 0015 正在实施数据迁移

具体迁移计划见 [`Feature 0015：Lesson 内容模型与听力库迁移`](../features/0015-lesson-content-model-migration/design.md)。

## 1. 目的

本文统一项目中与听力内容有关的产品术语、领域实体和资源边界，为后续课程制作、播客导入、有声书制作、中英双语音频、目录设计和代码命名提供共同语言。

本文主要解决以下问题：

1. 不再把每一份可播放内容笼统称为“课程”。
2. 使用一个统一实体表示从教材、Podcast、歌曲、视频、演讲和有声书等来源制作出的听力单元。
3. 区分内容组织、来源身份、音频文件、文字稿、翻译和播放进度。
4. 为未来把大型 MP3 迁移到 OSS 或 CDN 保留清晰边界。
5. 为同一内容的英文原声、中文 AI 配音、慢速音频等多个音频版本建立统一模型。
6. 为“英文三遍、中文一遍”等双语交叉播放模式建立可扩展概念。

本文是长期术语和目标模型约定，不表示当前代码、目录或 JSON Schema 已经完成迁移。当前行为仍以代码和编号较新的相关 Feature 文档为准。

## 2. 核心结论

项目使用 `Lesson` 表示最基本的可学习内容：

> **Lesson 是听力库中可独立打开、播放、下载和记录进度的基本学习单元。它拥有稳定的语义片段，并可以包含一种或多种音频呈现版本。**

一个 Lesson 可以来源于：

- 一篇教材课文；
- 一期 Podcast；
- 一首歌曲；
- 一段访谈；
- 一篇演讲；
- 一个 YouTube 视频；
- 一章有声书；
- 从长内容中制作出的独立精听部分。

Lesson 不是 MP3 文件。Lesson 是对标题、来源、音频、文字稿、翻译、说话人、授权信息和学习进度的统一包装。

项目的通用内容层级为：

```text
Library
└── Collection
    └── Lesson
```

其中 `Collection` 是通用组织容器，可以根据具体语义显示为课程、系列、播客、专辑、书籍或合集。

## 3. 规范术语

### 3.1 总体术语

| 英文术语 | 中文规范解释 | 定义 |
| --- | --- | --- |
| `Library` | 听力库 | 应用内所有 Lesson 和内容组织的总入口 |
| `Collection` | 内容集合 | 组织 Lessons 的通用容器 |
| `Course` | 课程 | 具有明确学习顺序或教学目标的一组 Lessons |
| `Series` | 系列 | 来源方持续发布或出版的一组内容 |
| `Playlist` | 播放列表 | 用户为了连续播放而组织的 Lesson 列表 |
| `Lesson` | 听力单元 | 可独立播放、学习、下载和记录进度的基本内容 |
| `Chapter` | 章节 | Lesson 内部用于导航的较大时间分段 |
| `Segment` | 字幕片段 | Lesson 中用于字幕显示和循环播放的最小稳定语义片段 |

### 3.2 内容资源术语

| 英文术语 | 中文规范解释 | 定义 |
| --- | --- | --- |
| `LessonMetadata` | Lesson 元数据 | 标题、类型、语言、时长等信息 |
| `SourceMetadata` | 来源信息 | 原始作品、作者、节目、网址、发布日期和授权信息 |
| `AudioAsset` | 音频资源 | 当前产品中的实际 MP3 媒体资源 |
| `AudioRendition` | 音频呈现版本 | 同一个 Lesson 的一种可选择音频版本 |
| `AudioSource` | 音频交付文件 | 某个 Rendition 对应的单个 MP3 文件及其对象 Key |
| `Transcript` | 文字稿 | 与 Lesson 内容对应的结构化原文 |
| `TimedTranscript` | 时间轴文字稿 | 带有开始和结束时间的 Transcript |
| `TranscriptSegment` | 文字稿片段 | 具有稳定 ID 的原文语义片段 |
| `Translation` | 简中译文 | 英文 TranscriptSegment 对应的 `zh-Hans` 翻译 |
| `Speaker` | 说话人 | 某个 TranscriptSegment 的发言者 |
| `LessonProgress` | Lesson 进度 | 用户在 Lesson 中的播放和学习位置 |
| `PlaybackPattern` | 播放模式 | 描述不同 Rendition 的播放顺序、次数和倍速 |
| `PlaybackSession` | 播放会话 | 用户本次打开播放器后产生的临时运行状态 |

### 3.3 中文使用约定

- 工程设计、Schema 和代码统一使用 `Lesson`。
- 中文规范文档第一次出现时可写作“Lesson（听力单元）”。
- 日常产品文案可以自然使用“一份听力材料”。
- 来源类型应使用自然名称，例如“播客单集”“歌曲”“教材课文”和“有声书章节”。
- 不要求用户界面把每个内容都显示为“一课”。

例如：

```text
工程实体：Lesson
来源类型：Podcast Episode
界面名称：播客单集
自然描述：一份听力材料
```

## 4. 内容组织

### 4.1 Library

`Library` 是所有内容的总入口，中文使用“听力库”。

Library 可以提供不同的展示分区，例如：

- 继续学习；
- 最近添加；
- 课程；
- Podcast；
- 有声书；
- 歌曲；
- 自选合集。

这些展示分区不一定是持久化的父子关系，也可以是根据元数据动态生成的 Shelf。

### 4.2 Collection

`Collection` 是 Lesson 的通用组织容器。推荐通过 `kind` 表达具体业务语义：

```ts
type CollectionKind =
  | "course"
  | "series"
  | "podcast"
  | "album"
  | "book"
  | "curated";
```

界面根据 `kind` 使用自然名称：

| `kind` | 界面名称 | 示例 |
| --- | --- | --- |
| `course` | 课程 | 新概念英语第一册 |
| `series` | 系列 | 一套连续演讲 |
| `podcast` | 播客 | Lenny’s Podcast |
| `album` | 专辑 | 一张音乐专辑 |
| `book` | 有声书 | Pride and Prejudice |
| `curated` | 合集 | YouTube 演讲精选 |

### 4.3 Course

`Course` 是一种具有学习顺序或教学目标的 Collection。

判断标准：

> 如果移除 Lesson 顺序会破坏学习设计，这个组织通常应当是 Course。

适合称为 Course 的例子：

- 新概念英语第一册；
- 30 天 Podcast 精听训练；
- 从入门到进阶的发音课程。

不应仅因为若干内容位于同一文件夹，就把它们称为 Course。

### 4.4 Series

`Series` 表示来源方定义的连续内容，不等同于教学课程。

例如：

```text
Series：Lenny’s Podcast
└── Lesson：OpenAI Codex lead on the new shape of product work
```

Podcast 平台中的 Show 在本项目的通用来源模型中可以表示为 `Series`，界面仍显示“播客”。

### 4.5 Playlist

Playlist 只表示用户希望按什么顺序播放内容，不表示内容归属。

```text
Collection / Course / Series
= 内容属于哪里

Playlist
= 用户接下来想播放什么
```

一个 Lesson 可以同时存在于来源 Series、编辑 Collection、学习 Course 和用户 Playlist 中。

### 4.6 多对多关系

Lesson 不应被物理目录限制为只能属于一个 Collection。

例如同一期 Lenny 访谈可以同时出现于：

```text
Series：Lenny’s Podcast
Collection：AI 与产品
Course：30 天播客精听
Playlist：通勤播放列表
```

这些位置引用同一个 Lesson ID、同一份 Transcript 和相同的音频资源。

## 5. 来源身份与学习身份

项目必须区分：

```text
内容原本是什么
与
内容进入学习系统后是什么
```

进入应用后，所有可独立学习的内容都是 Lesson；它原本的身份通过 `LessonKind` 和 `SourceMetadata` 保留。

```ts
type LessonKind =
  | "textbook"
  | "podcast"
  | "song"
  | "audiobook"
  | "video"
  | "speech"
  | "conversation"
  | "other";
```

### 5.1 新概念英语

```text
Library
└── Course：New Concept English Book 1
    ├── Lesson：001 Excuse me!
    ├── Lesson：003 Sorry, sir.
    └── Lesson：005 Nice to meet you
```

- “新概念英语”是教材系列。
- “第一册”在应用中可以作为 Course。
- 每份独立音频及其文字稿构成一个 Lesson。
- 第一册现有音频覆盖一课正文和下一课练习时，仍以一份独立音频形成一个 Lesson。

### 5.2 Podcast

```text
Library
└── Series：Lenny’s Podcast
    └── Lesson：OpenAI Codex lead on the new shape of product work
```

- 来源身份是 Podcast Episode。
- 应用实体是 Lesson。
- 一小时或更长的时长不会自动改变它的身份。

如果保留一份完整 MP3，它仍然是一个 Lesson，可以在内部增加 Chapters。

如果为了学习把原始音频实际拆成多个可独立下载、播放和记录进度的 MP3，则每一部分可以成为一个 Lesson，并保留共同的原始 Episode 来源信息。

### 5.3 歌曲

```text
Library
└── Album：Album Name
    └── Lesson：Song Title
```

歌曲在音乐来源中是 Track，进入学习系统后是 Lesson。`Track` 不作为本项目的通用学习实体，因为它还常用于表示媒体文件内部的音轨。

### 5.4 有声书

```text
Library
└── Book：Pride and Prejudice
    ├── Lesson：Chapter 1
    └── Lesson：Chapter 2
```

如果一章音频过长，也可以在保留来源章节信息的情况下制作多个 Lesson Parts。

### 5.5 YouTube 演讲

```text
Library
└── Collection：YouTube 演讲精选
    ├── Lesson：Steve Jobs’ Stanford Commencement
    └── Lesson：How Great Leaders Inspire Action
```

YouTube 只是来源平台。Lesson 应保存原始视频 ID、网址、作者、发布日期和授权状态，不把 YouTube 下载文件本身视为 Lesson。

## 6. Lesson 边界

一份内容通常满足以下条件时构成一个 Lesson：

1. 有稳定且全局唯一的 Lesson ID。
2. 可以独立进入播放器。
3. 可以独立记录学习进度。
4. 可以独立下载或删除缓存。
5. 有一组稳定的语义 Segment。
6. 至少有一个可播放的 AudioRendition。
7. 有与其语义内容对应的 Transcript。

“有时长”不是充分条件。Chapter、Segment、广告和静音区间也都有时长，但它们通常不是独立 Lesson。

### 6.1 Chapter 与 Lesson

Chapter 是 Lesson 内部导航点，不拥有独立下载和学习进度时，不应升级为 Lesson。

```text
Lesson：70 分钟 Podcast
├── Chapter：Introduction
├── Chapter：How product work is changing
└── Chapter：The role of engineers
```

### 6.2 Segment 与 Lesson

Segment 是字幕和循环播放的基本单位：

```text
Lesson
└── Transcript
    └── TranscriptSegment
```

Segment 可以被重复播放，但不单独出现在听力库，不拥有独立的 LessonProgress。

## 7. Lesson 的资源结构

目标概念结构为：

```text
Lesson
├── LessonMetadata
├── SourceMetadata
├── Speaker[]
├── Transcript
│   └── TranscriptSegment[]
├── Translation
├── AudioRendition[]
│   └── AudioSource
├── Chapter[]
└── LessonProgress
```

### 7.1 LessonMetadata

LessonMetadata 描述 Lesson 本身，通常包含：

- `id`；
- `kind`；
- 原始标题；
- 主要语言；
- 预计或实际时长；
- 图片；
- 难度；
- 英文 Transcript 和简体中文 Translation 数据；
- AudioRendition 定义。

LessonMetadata 不应包含：

- OSS 密钥；
- 本地绝对路径；
- Whisper 临时结果；
- 翻译工作流临时分块；
- 大型 MP3 二进制内容。

### 7.2 SourceMetadata

SourceMetadata 保存来源和授权判断所需信息：

- 来源类型；
- 原始作品或节目名称；
- 作者、主持人和嘉宾；
- 原始网址；
- YouTube ID、Podcast GUID 等来源标识；
- 发布日期；
- 出版方；
- 版权持有人；
- 音频、Transcript、翻译和改编的授权状态；
- 必须展示的署名。

公开链接只证明资源可以访问，不证明拥有再发布、翻译或配音的权利。

### 7.3 Transcript

Transcript 是原始语言的权威结构化文字稿。

每个 TranscriptSegment 至少包含：

```ts
interface TranscriptSegment {
  id: string;
  speakerId?: string;
  text: string;
}
```

在当前实现中，Segment 还直接包含原始音频的 `start` 和 `end`。目标中英 Rendition 模型把时间映射放到对应 AudioRendition 中。

Segment ID 必须稳定。翻译、音频时间轴和后续注释通过 Segment ID 关联，不应依赖数组下标。

### 7.4 Translation

当前产品只支持英文原文和简体中文译文。Translation 与英文 TranscriptSegment 放在同一个发布文档中，不再设计任意目标语言映射：

```json
{
  "id": "seg-000003",
  "text": "Thanks for having me.",
  "translation": {
    "text": "感谢邀请。"
  }
}
```

不推荐：

```json
{
  "start": 11.16,
  "end": 12.42,
  "english": "Thanks for having me.",
  "chinese": "感谢邀请。"
}
```

后者会复制原文和时间轴，容易在英文校对后产生漂移。

Lesson 文档记录 Transcript revision。翻译工作流的 source digest 用该 revision、Segment ID、英文、说话人和原声时间轴判断译文是否过期；发布文件不再维护一套容易漂移的跨文件 revision。

### 7.5 AudioAsset

AudioAsset 是可播放的实际媒体资源。它可以保存：

- 对象 Key；
- MIME 类型；
- 时长；
- 字节数；
- SHA-256；
- 编码信息。

应用数据应保存稳定的对象 Key，不把具体 OSS 域名写死在每一个 Lesson 中。

### 7.6 AudioRendition

AudioRendition 表示同一个 Lesson 的一种可选择音频呈现版本。当前只定义：

- 英文原声；
- 中文 AI 配音；

`AudioRendition` 是流媒体和媒体处理领域常用的内部工程术语。用户界面统一显示“音频版本”。

```ts
type LessonLanguage = "en" | "zh-Hans";
type AudioRenditionRole = "original" | "dub";
```

不为其他语言、评论音轨、无障碍描述或慢速版本预留当前产品接口；以后确有需求时再通过新的 Feature 扩展。

### 7.7 AudioSource

AudioRendition 是播放语义，AudioSource 是实际 MP3。两者在 `<lesson-id>.json` 内是不同字段，MP3 二进制仍位于独立音频目录：

```text
AudioRendition：中文 AI 配音
└── AudioSource：lenny-linear-product-zh-ai.mp3
```

当前每个 Rendition 只对应一个 MP3 AudioSource。英文原声与中文配音属于两个 Rendition，并各自拥有时间轴。

## 8. 目标发布 JSON

职责需要清晰，但发布文件不按职责机械拆分。每个 Lesson 只发布一个结构化 JSON，音频文件独立：

```text
content/lessons/lenny-linear-product.json
audio/lessons/lenny-linear-product-en.mp3
audio/lessons/lenny-linear-product-zh-ai.mp3
```

中文配音尚未制作时，第二个 MP3 和对应 Rendition 都可以不存在。英文 Transcript 必须存在；简体中文 Translation 按界面规则可选，缺失不得阻止播放。

### 8.1 为什么合并 JSON

- 一次请求和一次 Zod 解析即可得到可播放数据；
- Segment、译文和 Cue 在同一文档内校验，减少跨文件引用和 revision 漂移；
- 当前只支持固定的 `en` 与 `zh-Hans`，不存在按任意语言懒加载文件的收益；
- 数百 KB 的长 Podcast JSON 仍属于合理的单文件规模；
- 制作时仍可分块翻译，只有最终发布产物合并。

Collection 仍使用独立的 `<collection-id>.json`，因为它组织多个 Lesson；MP3 仍独立，因为它体积大、缓存和部署生命周期与 JSON 不同。三个发布目录都采用扁平文件结构。

## 9. 内容与音频的目录边界

推荐在目标架构中把可提交的小型内容数据与大型音频彻底分开：

```text
content/
├── catalog.json
├── collections/
└── lessons/

audio/
└── lessons/

content-work/
├── <lesson-id>/
└── _shared/
```

职责如下：

```text
content/collections
= Lessons 如何组织

content/lessons
= 可提交 Git 的完整 Lesson JSON

audio/lessons
= 不提交 Git、可迁移到 OSS 的大型媒体

content-work/<lesson-id>
= 一份 Lesson 的下载、转写、翻译、校对、报告和候选导出

content-work/_shared
```

## 10. 中英双语音频与交叉播放

### 10.1 同一 Lesson 的多个音频版本

一份英文 Podcast 可以拥有：

```text
Lesson
├── AudioRendition：en
└── AudioRendition：zh-ai
```

两个 Rendition 表达相同语义，但时长和时间轴可以不同。它们通过相同的 Segment ID 对齐。

```text
seg-000003
├── English text：Thanks for having me.
├── Chinese text：感谢邀请。
├── en：11.16 → 12.42
└── zh-ai：10.38 → 11.30
```

### 10.2 PlaybackPattern

“英文三遍、中文一遍”不应写死在播放器中，而应表示为 PlaybackPattern：

```json
{
  "id": "en-3-zh-1",
  "title": "英文 3 遍，中文 1 遍",
  "steps": [
    {
      "renditionId": "en",
      "repeats": 3,
      "playbackRate": 1
    },
    {
      "renditionId": "zh-ai",
      "repeats": 1,
      "playbackRate": 1
    }
  ]
}
```

同一模型可以支持：

```text
仅英文：
EN × 1

英中对照：
EN × 1 → ZH × 1

精听：
EN × 3 → ZH × 1

先理解后精听：
ZH × 1 → EN × 3

回译确认：
EN × 2 → ZH × 1 → EN × 1
```

### 10.3 播放状态

双语循环需要记录：

```ts
interface BilingualLoopState {
  segmentId: string;
  stepIndex: number;
  repeatIndex: number;
  renditionId: string;
}
```

状态推进必须使用同一处理路径和切换锁：

```text
当前 Cue 结束
    ↓
完成本次 repeat
    ↓
当前 step 是否还有重复？
├── 是：重播当前 Rendition 的 Cue
└── 否：进入下一个 step
          ↓
      是否还有 step？
      ├── 是：切换 Rendition
      └── 否：进入下一个 Segment
```

有限循环完成最后一个 Segment 后仍应暂停。

### 10.4 播放器实现约束

未来实现中英 Rendition 播放时继续遵守：

- 使用 `HTMLAudioElement`；
- 禁止通过 `decodeAudioData()` 解码完整 MP3；
- 禁止把一小时音频载入 `AudioBuffer`；
- 不为每个 Segment 创建独立的长期音频实例；
- 任意时刻只允许一个 Rendition 发声；
- Rendition 切换和原有循环共用切换锁；
- 页面导航不得销毁正在进行的播放会话；
- iPhone Safari、主屏幕 Web App、锁屏和后台切换必须进行真实设备验证。

可以由一个播放控制器管理持续存在的原声和配音 `HTMLAudioElement`，但 iOS 后台自动衔接能力必须先验证。若不可靠，可降级为：

- 前台支持动态中英文交叉；
- 后台只播放单个 Rendition；
- 或为固定模式预生成组合 MP3。

### 10.5 学习体验

双语交叉播放适合：

- 初级学习者快速确认句意；
- 长难句精听；
- 通勤时无法阅读字幕；
- 多说话人访谈；
- 重复后检查理解。

中文音频必须可关闭。中高级学习者可能希望避免频繁中文打断英语语流，因此产品应至少提供：

- 仅英文；
- 英中交叉；
- 自定义模式。

播放器可以持久化用户选择的 PlaybackPattern，但不持久化正在进行的循环会话。

## 11. AI 配音约定

中文 AI 配音应当：

1. 使用明确标注的合成声音。
2. 默认使用中性声音，不模仿主持人、嘉宾或其他真实人物。
3. 在元数据中记录 `synthetic: true`。
4. 记录所依据的 `transcriptRevision` 和翻译 source digest。
5. 在简中译文改动后把旧配音标记为过期。
6. 保存生成日期、声音类型和必要的可复现信息。
7. 在用户界面说明它不是来源方官方中文版本。

推荐文案：

```text
AI 生成中文配音
非原节目官方中文版本
```

适合阅读的译文和适合语音合成的脚本可能不同。`text` 保存准确显示译文，`speechText` 只在确有需要时保存口语化朗读脚本。

## 12. 授权与公开发布

每个 Lesson 在公开发布前必须分别判断：

- 原始音频是否允许再发布；
- Transcript 是否允许复制和发布；
- 翻译是否获得许可；
- 中文配音或其他改编是否获得许可；
- 封面和图片是否允许使用；
- 必须提供什么署名；
- 是否只能用于私人学习。

翻译和配音通常属于基于原作品的改编。使用中性 AI 声音只能降低声音仿制风险，不能替代原作品的翻译、改编和再发布授权。

授权不明确时：

- 可以继续本地制作、结构设计和技术验证；
- 不得把完整原声、完整 Transcript 或中文配音公开部署；
- `rights` 状态保持 `unverified`；
- 不得使用“公开可访问”推导“允许再发布”。

## 13. 数据版本和校验

所有外部 JSON 必须通过 Zod 校验。

### 13.1 LessonDocument 校验

- Lesson ID 格式合法且唯一；
- `kind` 属于允许值；
- 默认 Rendition 存在；
- `sourceLanguage` 固定为 `en`；
- `translationLanguage` 固定为 `zh-Hans`；
- AudioSource Key 不得是本地绝对路径；
- 权限状态属于已知枚举。

### 13.2 Segment 校验

- Segment ID 唯一；
- 原文非空；
- `speakerId` 引用已知 Speaker；
- revision 为有效正整数。

### 13.3 Translation 校验

- 译文非空；
- `speechText` 存在时非空；
- 允许 Segment 缺少 Translation，以兼容无翻译或制作中的 Lesson；
- 不接受 `zh-Hant`、`ja` 或任意动态语言键。

### 13.4 Rendition 校验

- Rendition ID 唯一；
- `language` 只能是 `en` 或 `zh-Hans`；
- `role` 只能是 `original` 或 `dub`；
- Cue 的 Segment ID 在 Transcript 中存在；
- `start >= 0`；
- `end > start`；
- Cue 不超过对应音频时长；
- AudioSource MIME 类型和文件格式一致；
- SHA-256 格式有效。

说话人抢话可能产生少量真实重叠，校验器应报告异常重叠，不宜无条件拒绝所有重叠。

## 14. 制作流程

正式 Lesson 与制作工作台必须分开：

```text
content-work/
├── lenny-andrew-ambrosino-2026/
│   ├── sources/
│   ├── interim/
│   ├── translation/
│   ├── migration/
│   ├── reports/
│   ├── exports/
│   └── tmp/
└── _shared/
    └── migration/
```

其中：

- `sources/` 保存下载或引用的原始资源；
- `interim/` 保存 ASR 输出、切分、格式转换和校对记录；
- `translation/` 保存分块翻译中间文件；
- `migration/` 保存该 Lesson 的格式迁移输入和过程文件；
- `reports/` 保存该 Lesson 的校验、SHA 和制作报告；
- `exports/` 保存等待验证并进入正式目录的候选结果；
- `tmp/` 只保存可随时删除的临时文件；
- `_shared/` 只保存确实跨多个 Lesson 的迁移、基线和汇总报告。

顶层优先使用稳定的 Lesson ID，而不是可变的播客标题或 Collection 路径。这样一期播客的全部制作材料保持聚合，
Lesson 移入其他 Collection 时也无需移动工作目录。

大型音频、下载文件和制作中间文件不得提交 Git。正式 Translation 继续使用 `scripts/translation-workflow.mjs` 的分块流程；合并阶段校验整个 `<lesson-id>.json` 后原子写入，不由人工直接批量重写大型 JSON。

## 15. 命名约定

### 15.1 ID

ID 应：

- 全局稳定；
- 使用小写 ASCII；
- 使用连字符分隔；
- 不依赖可变标题；
- 不因为移动 Collection 而改变。

示例：

```text
nce-1-001
lex-sam-altman-2025
lenny-andrew-ambrosino-2026
pride-and-prejudice-chapter-01
youtube-steve-jobs-stanford-2005
```

### 15.2 文件名

目标模型的 Lesson 文件名为：

```text
<lesson-id>.json
```

当前语言是固定产品约束：

```text
英文原文和原声：en
简体中文译文和配音：zh-Hans
```

Lesson JSON 不使用语言后缀，因为英文、简中和 Rendition 已在同一 `<lesson-id>.json` 内明确标识。MP3 文件名使用 `<lesson-id>-en.mp3` 和可选的 `<lesson-id>-zh-ai.mp3`；文件名全部使用小写 ASCII，JSON 内的语言标签仍使用标准 `zh-Hans`。

### 15.3 代码命名

推荐：

```text
Lesson
LessonId
LessonKind
LessonMetadata
LessonProgress
LessonSource
Collection
CollectionKind
AudioAsset
AudioRendition
AudioSource
Transcript
TranscriptSegment
Translation
PlaybackPattern
```

避免把完整 Lesson 命名为：

- `Audio`；
- `Track`；
- `Recording`；
- `Resource`；
- `Material`；
- `Episode`。

这些词可以作为资源或来源类型，但不能覆盖所有 Lesson。

## 16. 与当前实现的关系

当前实现仍使用 Feature 0010 确立的版本 1 JSON：

```json
{
  "version": 1,
  "course": {
    "id": "nce-3-001",
    "title": "Lesson 1: A Puma at Large",
    "audioFilename": "01－A Puma at large.mp3",
    "duration": 213.5,
    "language": "en"
  },
  "segments": [
    {
      "id": "segment-0001",
      "start": 0,
      "end": 4.2,
      "text": "Pumas are large, cat-like animals...",
      "translations": {
        "zh-Hans": "美洲狮是一种体形似猫的大型动物……"
      }
    }
  ]
}
```

现阶段：

- `course` 实际承担本文 `Lesson` 的职责；
- `segments` 同时承担 Transcript、时间轴和 Translation 的职责；
- `audioFilename` 指向单一原声音频；
- IndexedDB 仍使用现有 `course:<id>`、`audio:<id>` 和 `progress:<id>` 键；
- `src/lessons.ts` 的 Lesson 是课程目录项，不等于本文完整目标 Lesson Schema。

本文不授权直接重命名这些字段或迁移存储。

未来迁移必须通过单独 Feature 完成，并至少包括：

1. 新旧 Schema 的 Zod 校验；
2. 已下载旧 Lesson 的兼容读取；
3. `course:<id>` 等现有存储键的兼容策略；
4. 目录生成器更新；
5. `scripts/translation-workflow.mjs` 更新；
6. OPFS 和 IndexedDB 中英 Rendition 存储设计；
7. iPhone Safari 和主屏幕 Web App 验证；
8. 构建和部署脚本更新；
9. 音频 OSS 基础地址配置；
10. 回滚和数据迁移方案。

在该 Feature 实施前，不应为了术语统一而直接移动当前路径、改写全部 JSON 或恢复历史导入界面。

## 17. 决策摘要

本项目采用以下长期约定：

```text
Library
└── Collection
    └── Lesson
        ├── Transcript
        │   └── TranscriptSegment
        ├── Translation
        └── AudioRendition
            └── AudioSource
```

- `Lesson` 是跨教材、Podcast、歌曲、视频、演讲和有声书的统一学习实体。
- `Collection` 是 Lesson 的通用组织容器。
- `Course`、`Series`、`Podcast`、`Album` 和 `Book` 是具体 Collection 语义。
- `Playlist` 是播放安排，不是内容归属。
- Lesson 与 MP3 分离。
- 同一 Lesson 当前可以拥有英文原声和简体中文配音两个 AudioRendition。
- 用户界面把 AudioRendition 称为“音频版本”。
- 原声和配音各有独立时间轴，通过 Segment ID 对齐。
- 每个 Lesson 发布一个 `<lesson-id>.json`；逻辑职责分区不等于物理拆成多个 JSON。
- 双语交叉播放由 PlaybackPattern 描述。
- 内容 JSON、音频媒体和制作工作台使用不同存储边界。
- 未确认授权的原声、Transcript、翻译和配音不得公开部署。
- 当前 v1 Schema 保持不变，迁移必须另立 Feature。
